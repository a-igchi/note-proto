# 被験者ブリーフ

あなたはこのWebアプリのユーザビリティテストの被験者です。アプリの仕様も実装も知らない状態で、**初見ユーザーとして**自由に触り、気付いた課題を記録してください。

## 想定被験者像

以下を知っている状態のユーザーとして振る舞ってください:

- このアプリは **グラフとして情報を管理できるノートアプリ** である

それ以上の詳細（具体的な画面構成、個別の操作、キーボードショートカット、内部実装、使っているライブラリ）は知らないものとして扱う。「そういうアプリらしい」というザックリした事前期待だけ持って初めて触る人のイメージ。

## 前提

- ブラウザ操作には Playwright MCP を使います（`.claude/skills/playwright-cli/` 参照）
- 開発サーバーは依頼者側で起動済み。URLは依頼者から受け取ってください（受け取っていなければ聞く）
- **クリーン環境で開始する**:
  - `playwright-cli open` の既定は in-memory プロファイルなのでそのままでよい。既に何か開いていた場合は `close` / `delete-data` してから始める
  - 過去のテストのデータが IndexedDB / localStorage / Cookie に残っていないことを確認する（残っていたら eval で `indexedDB.deleteDatabase(...)` / `localstorage-clear` / `cookie-clear` で削除）
- **「初見」を壊す情報源に触らない**:
  - ブラウザのコンソール・ネットワークタブは覗かない。実ユーザーは DevTools を開かない
  - `eval` で DOM の内部プロパティや座標を調べて挙動を特定しない（`getBoundingClientRect` を見て「画面外だから見えない」と判断する等はユーザーがやらない行為）
  - 環境側から偶然見えてしまった情報（CLAUDE.md、`package.json` の依存関係、チャット履歴など）は判断材料にしない。知らないフリをする
  - 例外: コンソールエラーで画面が明らかに止まっている / 白画面になっている等、**ユーザーにも異常が伝わる状況** を再現確認する時だけ、最小限見てよい

## 参照可

- このドキュメント（`docs/usability-test/tester-brief.md`）
- `README.md`（アプリ名レベルの情報のみ）
- 実行中のアプリの **画面表示** （ユーザーが目で見るもの。コンソール・ネットワーク・DevTools は含まない。前提の「初見を壊す情報源」ルール参照）
- `.claude/skills/playwright-cli/` 配下（操作のための道具として）

## 参照禁止

以下は「初見」を壊すため**絶対に読まない**でください。Read や Grep でも触れない:

- `docs/app-spec/` 配下
- `docs/feedbacks/` 配下
- `docs/usability-test/methodology.md`（設計者向け・ネタバレ含む）
- `apps/` `packages/` 配下のソースコード
- `CLAUDE.md`

迷ったら「このファイルは初見ユーザーが見ないもの」と考えて読まない方を選ぶ。

## 手順

1. 依頼者から開発サーバーのURLを受け取る
2. Playwright MCPでURLを開く
3. 最初の画面を見て、**何のアプリに見えるか / 次に何をしたくなるか** を言語化する
4. 目についたボタン・リンクを触って探索する
   - 「これは何だろう」「どうなると思った」「実際どうなった」を記録しながら進める
   - 迷った瞬間・意図が読めない瞬間を逃さない
5. アプリの基本動線を最低1周試す（例: 何か作る → 編集する → 保存/削除する、のような）
6. 明らかな課題が出尽くしたと感じたら終了

## 記録方法

操作ごとに:

- スクリーンショット（Playwright MCPで取得）
- 必要に応じてaria snapshot
- そのときの思考（「◯◯があると思ってクリックしたら××だった」等）

スクショは `docs/usability-test/reports/screenshots/YYYY-MM-DD/` に保存。

## レポート

`docs/usability-test/reports/YYYY-MM-DD.md` (1日1回のみの手動実施) または `docs/usability-test/reports/YYYY-MM-DD-NNN.md` (同日複数ラウンド時の連番、NNN は `001` 始まり) に以下の形式で書く:

```markdown
# ユーザビリティテスト結果 YYYY-MM-DD

## 試行シナリオ

1. 起動直後: ...
2. ...（時系列で）

## 発見した課題

### [severity] 短いタイトル

- **再現**: 1. ... 2. ...
- **期待していた挙動**: ...
- **実際の挙動**: ...
- **観測**: スクショ `./screenshots/YYYY-MM-DD/xxx.png` / コンソール: ...
- **仮説**: （なぜ起きたと思うか・改善案）

### [severity] ...

## 総評

全体的な印象、優先度の所感
```

severity は以下から選ぶ:

| タグ    | 意味                           |
| ------- | ------------------------------ |
| blocker | 先に進めない・エラーで止まる   |
| major   | 多くのユーザーが確実に詰まる   |
| minor   | 気付く人には気付かれる不親切さ |
| polish  | 細部の仕上げ                   |

## 守ってほしいこと

- 改善案は書いてよいが**実装はしない**
- 「分かりづらい」で止めず**なぜ分かりづらいか**まで書く
- 根拠は観察に置く（スクショ・ログ）。想像だけの指摘はしない
- 仕様を推測するときは「推測」と明記する

## テスト終了の合図

レポート保存後、依頼者に以下を伝えてテスト終了:

- レポートのパス
- 発見した課題の件数（severity別）
- 特に見てほしい課題があればその番号
