# ユーザビリティテスト方針

このドキュメントはテストの設計者・レビュー者向け。被験者（テスト実施者）は読まない。

## 目的

開発者の盲点になっているUI/UX上の課題を、AIエージェントを「初見ユーザー」に見立てて探索的に洗い出す。仮説検証ではなく発見的テスト。

## 手法

- Exploratory usability testing（タスク事前指定なし）
- 被験者: アプリの仕様も実装も知らない新規Claude Codeセッション
- ブラウザ操作: Playwright MCP（`.claude/skills/playwright-cli/` のskillを使用）

## セッション分離

被験者は必ず**新規セッション**で起動する。現行セッションの文脈を引き継ぐと「初見」が成立しない。

## 被験者の情報アクセス制限

### 参照可

- `docs/usability-test/tester-brief.md`
- `README.md`（アプリ名レベル）
- 実行中のアプリそのもの（Playwright経由の操作結果、コンソール、ネットワーク）

### 参照禁止

以下は「初見の視点」を壊すため参照させない:

- `docs/app-spec/` 配下
- `docs/feedbacks/` 配下
- `apps/` `packages/` 配下のソースコード
- `CLAUDE.md`、その他開発者向けドキュメント

tester-brief.md 側にもこの禁止リストを明記してある。

## 事前セットアップ（依頼者が実施）

被験者セッション起動前に、依頼者側で以下を済ませる:

1. `vp install`
2. `vp run dev` で開発サーバー起動
3. 起動URLを控え、被験者に伝える（またはtester-briefに追記）

被験者にセットアップを任せない理由: コマンド探索中に他ドキュメントを覗いてしまうリスクを避けるため。

## 記録項目

被験者に以下を逐次記録させる:

- 画面ごとの第一印象（「何のアプリに見えるか」「次に何をしたくなるか」）
- 意図が読めなかったUI要素
- 迷った操作（クリックしてみたが期待と違った等）
- 予想と異なった挙動
- コンソールエラー・壊れた表示・ネットワーク失敗
- 各操作時点のスクリーンショットとaria snapshot

## 成果物

`docs/usability-test/reports/YYYY-MM-DD.md` に保存。

構成:

- 試行シナリオ（時系列）
- 発見した課題（severity タグ付き: blocker / major / minor / polish）
- 各課題: 再現手順・観測結果（スクショ・ログ）・改善の仮説
- 総評

スクショは `docs/usability-test/reports/screenshots/YYYY-MM-DD/` 配下。

## severity の定義

| タグ    | 定義                                           |
| ------- | ---------------------------------------------- |
| blocker | 機能が成立しない、エラーで止まる、先に進めない |
| major   | 回避はできるが多くのユーザーが確実に詰まる     |
| minor   | 気付く人には気付かれる不親切さ                 |
| polish  | 細部の仕上げ・整合性                           |

## 留意点

- 被験者は改善案を書いてよいが**実装はしない**（別タスクで扱う）
- 「分かりづらい」で止めず「なぜ分かりづらいか」まで言語化させる
- 根拠は観察（スクショ・ログ）に置く。思い込みの指摘は避ける
- 1セッション1レポート。追試は別レポートとして分けて比較可能にする

## このテストで得られないもの

以下は本手法の範囲外。必要なら別手段で補う:

- 主観的な心地よさ・美しさ評価
- 発話思考法的な認知負荷測定
- 長期利用での学習曲線
- 複数ユーザー間の認知差
