# アーキテクチャ設計

## 概要

本ドキュメントでは、アプリケーションのアーキテクチャについて説明する。

## 全体構成

サーバーとフロントエンドから成るSPA構成をとる。

```
[ブラウザ]
    │
    │  HTTP (REST API / 静的ファイル配信)
    │
[サーバー (Hono + Node.js)]
    │
    ├── SQLite (メタデータ・リンク)
    └── ファイルシステム (ノート本文)
```

## 開発環境

| 項目    | バージョン |
| ------- | ---------- |
| Node.js | 22以上     |
| pnpm    | 10以上     |

pnpmワークスペースで管理する。

```
pnpm install          # 依存のインストール
pnpm --filter server dev   # サーバー起動
pnpm --filter web dev      # フロントエンド起動
```

## パッケージ構成

モノレポ構成で、以下のパッケージに分割する。

| パッケージ    | 役割              |
| ------------- | ----------------- |
| packages/core | 型定義            |
| apps/server   | HTTPサーバー      |
| apps/web      | Webフロントエンド |

### core

サーバーとフロントエンドが共有する型定義を提供する。[API設計](api.md)の型定義セクションを参照。

## サーバー

### フレームワーク

Hono を採用する。軽量なWebフレームワークで、Node.js上で動作する。

### ミドルウェア

- **CORS**: `/api/*` に対してCORSを有効にする。開発時にフロントエンドとサーバーが異なるポートで動作するため。

### ルーティング

```
/api/notes    → notes ルーター
/api/links    → links ルーター
/api/graph    → graph ルーター
```

各ルーターの詳細は [API設計](api.md) を参照。

### データアクセス

SQLiteは `node:sqlite` の同期API (`DatabaseSync`) を使用する。データベースの初期化（テーブル作成）はサーバー起動時に実行する。

ファイルシステムへのアクセスは `node:fs` の同期APIを使用する。

## フロントエンド

### フレームワーク・ビルドツール

- **React**: UIライブラリ
- **Vite**: ビルドツール・開発サーバー

### ライブラリ

#### UIコンポーネント: shadcn/ui

UIコンポーネントには shadcn/ui を採用する。Radix UIプリミティブの上にTailwind CSSでスタイルを当てたコンポーネント集で、プロジェクトにコードとしてコピーされる（npmパッケージとしてインストールするのではなく、ソースコードを直接保持する）。

関連ライブラリ:

| ライブラリ               | 用途                                                  |
| ------------------------ | ----------------------------------------------------- |
| Radix UI                 | ヘッドレスUIプリミティブ（Dialog, Popover, Select等） |
| Tailwind CSS             | ユーティリティファーストCSS                           |
| class-variance-authority | コンポーネントのバリアント管理                        |
| tailwind-merge           | Tailwindクラスの競合解決                              |
| clsx                     | 条件付きクラス名の結合                                |
| Lucide React             | アイコンセット                                        |

#### エディタ: Milkdown

ノート本文のMarkdownエディタにはMilkdownを採用する。ProseMirrorベースのWYSIWYG Markdownエディタで、Markdownをリアルタイムにレンダリングしながら編集できる。

使用するプラグイン:

| プラグイン             | 用途                                    |
| ---------------------- | --------------------------------------- |
| @milkdown/kit          | コアエディタ + 基本的なMarkdownサポート |
| @milkdown/plugin-prism | コードブロックのシンタックスハイライト  |
| @milkdown/theme-nord   | テーマ                                  |

#### グラフレイアウト: d3-force

グラフのレイアウト計算にd3-forceを使用する。描画はReact SVGで行い、d3-forceはレイアウトエンジンとしてのみ利用する。d3のDOM操作機能は使用しない。詳細は後述の「グラフ可視化」を参照。

#### 状態管理: TanStack Query

TanStack Query（React Query）を使用する。サーバーの状態をキャッシュし、mutation後にキャッシュを無効化して再取得する。

**クエリキー設計:**

| キー           | 用途                   |
| -------------- | ---------------------- |
| `["graph"]`    | グラフデータ           |
| `["notes"]`    | ノート一覧             |
| `["note", id]` | 個別ノート（本文含む） |

**Invalidation戦略:**

ノート・リンクの作成・更新・削除時に `["graph"]` と `["notes"]` を無効化する。個別ノートのキャッシュは必要に応じて無効化する。

### 画面遷移

アプリケーションはノート一覧画面とノート詳細画面の2画面で構成される。React Routerを使用し、SPA内でルーティングを行う。詳細は[画面設計](view.md)を参照。

| パス         | 画面                             |
| ------------ | -------------------------------- |
| `/`          | ノート一覧画面（グラフ）         |
| `/notes/new` | ノート詳細画面（新規作成）       |
| `/notes/:id` | ノート詳細画面（既存ノート編集） |

新規作成時はフロントエンド上で編集し、戻る操作時に初めてAPIにPOSTする。

### コンポーネント構成

[画面設計](view.md)の画面名との対応を括弧内に示す。

```
App
├── GraphView          # ノート一覧画面（グラフの可視化、全画面SVG）
│   └── コンテキストメニュー（ノード: 削除・リンク追加、エッジ: 削除）
├── EditorPanel        # ノート詳細画面（タイトル + 本文）
└── LinkPicker         # リンク追加パレット
```

### グラフ可視化

d3-forceでレイアウトを計算し、React SVGで描画する。

**d3-forceのパラメータ:**

| Force   | 種類          | パラメータ     |
| ------- | ------------- | -------------- |
| link    | forceLink     | distance: 80   |
| charge  | forceManyBody | strength: -200 |
| center  | forceCenter   | キャンバス中央 |
| collide | forceCollide  | radius: 30     |

**シミュレーションの最適化:**

- ノードの追加・削除時: シミュレーションを再構築する。既存ノードの位置は引き継ぐ。
- リンクのみ変更時: シミュレーションを再構築せず、リンクフォースのみ更新して `alpha(0.3)` で再開する。

### エラーハンドリング

API呼び出しが失敗した場合、トースト通知でユーザーにエラーを表示する。自動リトライは行わない。

### データフロー

1. ユーザーがUIを操作する。
2. React Queryのmutationがfetch APIを呼び出す。
3. サーバーがSQLiteとファイルシステムを更新する。
4. Mutation成功時にReact Queryのキャッシュを無効化する。
5. 無効化されたクエリが自動で再取得される。
6. コンポーネントが新しいデータで再描画される。

### 自動保存

ノート本文の編集は、入力から500ms後に自動保存する（デバウンス）。タイトルの変更はフォーカスが外れたタイミングで保存する。
